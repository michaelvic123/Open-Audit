================================================================================
3-TIER VALIDATION ARCHITECTURE
================================================================================

TIER 1: JSON Schema Validation
-------------------------------
Tool: AJV (Another JSON Schema Validator)
Purpose: Structural integrity and type checking
Validates:
  ✓ Required fields present (contract_id, topics, event_structure, etc.)
  ✓ Correct data types (string, array, object)
  ✓ Contract ID format (56-char uppercase alphanumeric)
  ✓ Field naming conventions (snake_case)
  ✓ Type enums (address, i128, u32, symbol, bool, bytes, string)

TIER 2: Template Variable Cross-Examination
--------------------------------------------
Tool: Custom regex-based parser in lint-registry.ts
Purpose: Ensure template variables match event structure
Process:
  1. Extract all {bracketed} variables from english_template
  2. Get available fields from event_structure.topics and .data
  3. Compare extracted variables against available fields
  4. Report any mismatches with clear error messages

Example:
  Template: "Transfer {value} from {from} to {to}"
  Available: [from, to, amount]
  Error: Template references '{value}' but event_structure only provides
         [from, to, amount]

TIER 3: Logical Consistency Checks
-----------------------------------
Tool: Custom validation logic in lint-registry.ts
Purpose: Catch business logic errors
Validates:
  ✓ Topics array length = event_structure.topics.length + 1
  ✓ No duplicate entries (same contract_id + topics combo)
  ✓ test_vectors params match defined fields
  ✓ All test_vectors params are in event_structure

================================================================================
GITHUB ACTIONS INTEGRATION
================================================================================

Workflow: .github/workflows/registry-lint.yml
Triggers:
  - Pull requests modifying lib/translator/registry.json
  - Pull requests modifying lib/translator/registry.schema.json
  - Pushes to main/master branch (same paths)

Path Filtering:
  Only runs when translation files change (not on docs/code changes)
  Saves CI runner minutes

Steps:
  1. Checkout repository
  2. Setup Node.js v20
  3. Install dependencies
  4. Run Tier 1 validation (schema)
  5. Run Tier 2 & 3 validation (advanced)
  6. Generate job summary
  7. Comment on PR if failed

Fail-Fast Behavior:
  - Any validation failure stops the workflow
  - PR cannot be merged until checks pass
  - Clear error logs in Actions tab
  - Bot comments on PR with fix instructions

USAGE INSTRUCTIONS
================================================================================

FOR CONTRIBUTORS:
-----------------
1. Before committing registry changes:
   npm run lint:registry

2. If errors appear, fix them:
   - Read the error message carefully
   - Find the entry by index number
   - Fix the mismatch/error
   - Re-run linter

3. Push changes and open PR
   - GitHub Actions will run automatically
   - Check the Actions tab if validation fails
   - Green checkmark means ready for review

FOR MAINTAINERS:
----------------
1. Review PR as normal
2. Check that registry-lint workflow passed
3. Validation guarantees structural correctness
4. Still review for business logic and naming conventions
5. Merge when ready

FOR DEVELOPERS:
---------------
# Run validation locally
npm run lint:registry

# Run only schema validation (faster)
npm run validate:registry

# Test against bad examples
copy lib\translator\registry.bad-example.json lib\translator\registry.json
npm run lint:registry

# Restore original
git checkout lib/translator/registry.json
