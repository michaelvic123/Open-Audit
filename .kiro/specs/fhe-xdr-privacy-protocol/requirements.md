# Requirements Document

## Introduction

Open-Audit's FHE XDR Privacy Protocol enables institutional clients operating private Soroban networks to use the Open-Audit Translation Registry without exposing the actual values of their transactions to the Open-Audit backend servers. The protocol encrypts numeric value fields (amounts, balances) inside Stellar XDR events client-side in the browser using Fully Homomorphic Encryption (FHE). The Translation Registry backend applies structural transformations to ciphertexts homomorphically — it never decrypts, and cannot infer plaintext values. Decryption happens exclusively in the client's browser after the translated response is returned. Structural fields such as contract IDs, event type discriminants, and topic names remain in plaintext so that the existing blueprint matching and translation pipeline is unaffected.

This feature adds a new opt-in privacy mode to Open-Audit. It does not replace the existing public translation flow.

---

## Glossary

- **FHE_Client**: The in-browser TypeScript/WASM module responsible for key generation, value encryption, and result decryption. Runs exclusively in the end user's browser.
- **FHE_Context**: The serializable public encryption context (TFHE parameter set + public key) that the FHE_Client generates and shares with the backend.
- **Ciphertext**: An encrypted representation of a single numeric XDR value (e.g., a transfer amount or balance). Produced by the FHE_Client; opaque to the backend.
- **Translation_Registry**: The existing Open-Audit backend engine (`lib/translator/registry.ts`) extended with homomorphic evaluation support. Applies structural transformations to ciphertexts without decrypting them.
- **XDR_Value_Field**: A numeric field within a Stellar XDR event payload that carries financial meaning — specifically i128/u128 amounts, balances, and similar quantities. Does NOT include addresses, topic discriminants, or event-name symbols.
- **FHE_Envelope**: The wire format sent from the browser to the Translation_Registry. Contains the original XDR event with XDR_Value_Fields replaced by Ciphertexts, plus the FHE_Context and metadata.
- **FHE_Result**: The wire format returned by the Translation_Registry. Contains structurally-transformed Ciphertexts (not plaintexts) and a human-readable translation template with ciphertext placeholders.
- **TFHE_Library**: The WASM-compiled FHE library used in the browser. The recommended candidate is Zama's `tfhe` npm package (WASM build); the requirement is that it supports integer bootstrapped ciphertext operations (addition, comparison, conditional select) on i64/u64 values in a browser environment.
- **Homomorphic_Evaluator**: The server-side Node.js module that applies allowed arithmetic/comparison operations to Ciphertexts using only the FHE_Context (public key), never the secret key.
- **Privacy_Mode**: An opt-in session flag set by the institutional client to activate FHE encryption for their analysis session. When Privacy_Mode is inactive, the standard unencrypted translation pipeline is used unchanged.
- **Key_Pair**: The FHE secret key and corresponding public key generated by the FHE_Client. The secret key never leaves the browser.
- **Translation_Template**: A human-readable string with `{{ct_N}}` placeholders where encrypted values will be decrypted and inserted by the FHE_Client on the return path. Example: `"{{ct_0}} XLM transferred from {{addr_from}} to {{addr_to}}"`.

## Requirements

### Requirement 1: FHE Library Integration (Browser, WASM)

**User Story:** As an institutional client, I want the Open-Audit browser client to load a WebAssembly-compiled FHE library, so that cryptographic operations run in-browser without transmitting sensitive values to any server.

#### Acceptance Criteria

1. THE FHE_Client SHALL load the TFHE_Library as a WASM module during Privacy_Mode session initialization, before any XDR data is processed.
2. WHEN the TFHE_Library WASM module fails to load or initialize, THE FHE_Client SHALL surface a descriptive error to the UI and abort Privacy_Mode activation without processing any XDR data.
3. THE FHE_Client SHALL initialize within 10 seconds of the browser invoking Privacy_Mode activation on a device meeting the minimum specification (Chrome 120+, 8 GB RAM, WebAssembly SIMD support).
4. THE FHE_Client SHALL execute all cryptographic operations (key generation, encryption, decryption) within a Web Worker to prevent blocking the main browser thread.
5. WHEN Privacy_Mode is inactive, THE FHE_Client SHALL not load or initialize the TFHE_Library WASM module, so that no cryptographic overhead is imposed on standard sessions.

---

### Requirement 2: Client-Side Key Generation

**User Story:** As an institutional client, I want the FHE key pair to be generated entirely in my browser, so that my secret key is never transmitted to or stored on any Open-Audit server.

#### Acceptance Criteria

1. WHEN the institutional client activates Privacy_Mode, THE FHE_Client SHALL generate a Key_Pair using the TFHE_Library parameter set configured for integer operations on values up to 64-bit unsigned integers.
2. THE FHE_Client SHALL store the secret key exclusively in browser memory (not in localStorage, sessionStorage, IndexedDB, or cookies) for the lifetime of the Privacy_Mode session.
3. WHEN the Privacy_Mode session ends or the browser tab is closed, THE FHE_Client SHALL securely zero the secret key from memory before deallocation.
4. THE FHE_Client SHALL complete Key_Pair generation within 30 seconds on a device meeting the minimum specification defined in Requirement 1, Criterion 3.
5. THE FHE_Client SHALL serialize the public key component of the Key_Pair into the FHE_Context structure for transmission to the Translation_Registry.
6. IF Key_Pair generation fails due to insufficient browser resources or a WASM error, THEN THE FHE_Client SHALL return a structured error containing the failure reason and SHALL NOT proceed with FHE_Envelope construction.

---

### Requirement 3: XDR Value Field Identification

**User Story:** As an institutional client, I want only the numeric value fields (amounts, balances) within XDR events to be encrypted, so that the Translation Registry can still match event types and apply structural transformations without needing decryption.

#### Acceptance Criteria

1. THE FHE_Client SHALL identify XDR_Value_Fields within a RawEvent by inspecting the ScVal type discriminants of `topics` and `data` fields, selecting only entries whose decoded ScVal type is one of: `I128`, `U128`, `I64`, `U64`, `I32`, `U32`.
2. THE FHE_Client SHALL leave the following fields in plaintext within the FHE_Envelope: the event `id`, `contractId`, `ledger`, `timestamp`, `txHash`, and all topic entries whose decoded ScVal type is `Symbol`, `Address`, `String`, or `Bytes`.
3. WHEN a RawEvent contains no XDR_Value_Fields, THE FHE_Client SHALL include the event unmodified in the FHE_Envelope with an `encryptedFields: []` annotation, and THE Translation_Registry SHALL process it using the standard plaintext translation path.
4. THE FHE_Client SHALL produce an `encryptedFields` index within the FHE_Envelope whenever XDR_Value_Fields are detected in the RawEvent, mapping each Ciphertext (if any) to its source location in the original RawEvent (field name and array index). When fields are detected but none are ultimately encrypted, the index SHALL be present and empty.
5. THE FHE_Client SHALL parse XDR field types using the `stellar-sdk` XDR utilities already present in the project, without introducing a separate XDR parsing dependency.

---

### Requirement 4: Client-Side Value Encryption

**User Story:** As an institutional client, I want each numeric XDR value field to be individually encrypted as a Ciphertext before leaving my browser, so that the backend server sees only ciphertexts during analysis.

#### Acceptance Criteria

1. WHEN the FHE_Client identifies an XDR_Value_Field in a RawEvent, THE FHE_Client SHALL encrypt the decoded integer representation of that field using the TFHE_Library with the Key_Pair public key, producing a Ciphertext.
2. THE FHE_Client SHALL encode each Ciphertext as a Base64 string for inclusion in the FHE_Envelope JSON payload.
3. THE FHE_Client SHALL encrypt each XDR_Value_Field independently so that a Ciphertext for one field does not reveal information about other fields in the same event.
4. THE FHE_Client SHALL complete the encryption of all XDR_Value_Fields in a single RawEvent within 5 seconds per event on a device meeting the minimum specification.
5. IF the integer value of an XDR_Value_Field exceeds the supported plaintext modulus of the TFHE_Library configuration, THEN THE FHE_Client SHALL split the value into multiple Ciphertexts according to a documented decomposition scheme and SHALL record the decomposition metadata in the `encryptedFields` index.
6. THE FHE_Client SHALL include a deterministic nonce or counter in the FHE_Envelope to allow the Translation_Registry to detect and reject replayed FHE_Envelopes within a 5-minute window.

---

### Requirement 5: FHE Envelope Construction and Transmission

**User Story:** As an institutional client, I want the browser to assemble and transmit a well-formed FHE_Envelope to the Translation Registry, so that the backend receives everything it needs to perform homomorphic evaluation without any plaintext values.

#### Acceptance Criteria

1. THE FHE_Client SHALL construct an FHE_Envelope containing: the FHE_Context (serialized public key and parameter set identifier), the modified RawEvent (with XDR_Value_Fields replaced by Ciphertext Base64 strings), the `encryptedFields` index, a session identifier, and a UTC timestamp.
2. THE FHE_Client SHALL transmit the FHE_Envelope to the Translation_Registry via an HTTPS POST request to a dedicated `/api/v1/fhe/translate` endpoint, separate from the standard translation API.
3. THE FHE_Envelope SHALL be serialized as JSON with a maximum payload size of 10 MB per event. IF the serialized FHE_Envelope exceeds 10 MB, THEN THE FHE_Client SHALL return an error to the UI before transmission.
4. THE FHE_Client SHALL not transmit the FHE_Envelope over unencrypted HTTP. IF the runtime detects a non-HTTPS origin in a production build, THEN THE FHE_Client SHALL refuse transmission and surface an error.
5. THE FHE_Client SHALL include a `Content-Type: application/json` header and an `X-FHE-Protocol-Version` header set to the current protocol version on all FHE_Envelope requests.

---

### Requirement 6: Translation Registry Schema for FHE-Compatible Transformations

**User Story:** As an Open-Audit developer, I want the Translation Registry blueprint schema to declare which arithmetic and comparison operations are permitted on each value field, so that the Homomorphic_Evaluator can apply only pre-approved, structurally safe transformations.

#### Acceptance Criteria

1. THE Translation_Registry blueprint schema (currently `lib/translator/registry.json` and `registry.schema.json`) SHALL be extended with an optional `fhePolicy` section per event definition that enumerates the allowed homomorphic operations for each XDR_Value_Field slot: `add`, `sub`, `compare_eq`, `compare_lt`, `compare_gt`, `conditional_select`.
2. WHEN an `fhePolicy` section is absent from a blueprint, THE Translation_Registry SHALL treat all Ciphertext slots in that blueprint as passthrough (no arithmetic transformation applied), return the Ciphertexts unmodified in the FHE_Result, and respond with HTTP 200.
3. THE Translation_Registry SHALL validate each requested homomorphic operation against the `fhePolicy` for the matched blueprint before executing it. IF a requested operation is not listed in `fhePolicy`, THEN THE Translation_Registry SHALL reject the FHE_Envelope with HTTP 422 and a descriptive error, without performing any computation on the Ciphertexts.
4. THE Translation_Registry SHALL produce a Translation_Template string alongside each FHE_Result, replacing encrypted numeric slots with `{{ct_N}}` placeholders and keeping structural text (event type, addresses, labels) in plaintext.
5. THE Translation_Registry blueprint schema version SHALL be incremented to reflect the addition of the `fhePolicy` field, and the JSON Schema validator at `registry.schema.json` SHALL enforce the `fhePolicy` structure when present.

---

### Requirement 7: Backend Homomorphic Evaluation

**User Story:** As an institutional client, I want the Translation Registry backend to apply its structural transformations to my encrypted values without ever decrypting them, so that actual transaction amounts never exist in plaintext on the Open-Audit servers.

#### Acceptance Criteria

1. THE Homomorphic_Evaluator SHALL accept an FHE_Envelope, extract the FHE_Context, and perform all arithmetic/comparison operations on the Ciphertexts using only the public key material in the FHE_Context — never using a secret key.
2. THE Homomorphic_Evaluator SHALL process only operations explicitly listed in the matched blueprint's `fhePolicy`. THE Homomorphic_Evaluator SHALL not apply operations absent from `fhePolicy` even if they are computationally valid.
3. WHEN the FHE_Context public key fails validation against the TFHE_Library parameter set expected by the server, THE Homomorphic_Evaluator SHALL return HTTP 400 with an error code `FHE_CONTEXT_MISMATCH` and SHALL NOT process the Ciphertexts.
4. THE Homomorphic_Evaluator SHALL complete homomorphic evaluation of all Ciphertext operations for a single FHE_Envelope within 30 seconds per event on the server hardware. IF evaluation exceeds this threshold, THE Homomorphic_Evaluator SHALL return HTTP 504 with error code `FHE_EVAL_TIMEOUT`.
5. THE Homomorphic_Evaluator SHALL never log, persist, or transmit Ciphertext values to any store that persists beyond the lifetime of the HTTP request. All intermediate Ciphertexts SHALL be discarded from server memory upon response completion.
6. THE Homomorphic_Evaluator SHALL return an FHE_Result containing: the Translation_Template, the transformed Ciphertext(s) as Base64 strings indexed by the `encryptedFields` positions, and a server-side evaluation metadata object (operations applied, blueprint matched, schema version).

---

### Requirement 8: Client-Side Decryption and Result Rendering

**User Story:** As an institutional client, I want the decrypted transaction values to be rendered in my browser only after the FHE_Result is received, so that plaintext amounts are never present outside my browser session.

#### Acceptance Criteria

1. WHEN the FHE_Client receives an FHE_Result, THE FHE_Client SHALL decrypt each Ciphertext in the FHE_Result using the secret key held in browser memory, producing the transformed plaintext integer value.
2. THE FHE_Client SHALL substitute each decrypted value into the corresponding `{{ct_N}}` placeholder in the Translation_Template to produce the final human-readable description string.
3. THE FHE_Client SHALL validate that the number of Ciphertexts in the FHE_Result matches the number of `{{ct_N}}` placeholders in the Translation_Template, treating a count of zero on both sides as a successful match. IF the counts are non-zero and do not match, THE FHE_Client SHALL display an error in the dashboard and SHALL NOT render a partial description.
4. THE FHE_Client SHALL complete decryption and template substitution for a single FHE_Result within 5 seconds on a device meeting the minimum specification.
5. THE FHE_Client SHALL not write decrypted plaintext values to the browser's localStorage, sessionStorage, IndexedDB, or any persistent cache. Decrypted values SHALL exist only in React component state for the duration of the UI render cycle.
6. WHEN Privacy_Mode is active, THE Dashboard SHALL display a visual indicator (lock icon and "Privacy Mode" label) on each event row that was processed via FHE, distinguishing it from standard translated events.

---

### Requirement 9: Round-Trip Integrity

**User Story:** As an institutional client, I want to be able to verify that encrypt → homomorphic evaluate → decrypt produces the correct result, so that I can trust that the FHE protocol has not silently corrupted transaction values.

#### Acceptance Criteria

1. THE FHE_Client SHALL provide a `verifyRoundTrip` function that: encrypts a known test integer, transmits it to the Translation_Registry via the FHE endpoint with a no-op passthrough blueprint, decrypts the returned Ciphertext, and asserts that the decrypted value equals the original test integer.
2. WHEN `verifyRoundTrip` is invoked, THE FHE_Client SHALL complete the full round trip and return a pass/fail result within 60 seconds.
3. FOR ALL valid integer values in the range [0, 2^63 - 1], the result of `FHE_Client.decrypt(FHE_Client.encrypt(v))` SHALL equal `v` (identity round-trip property, verified by the client-side test suite).
4. THE Translation_Registry SHALL expose a dedicated health endpoint `GET /api/v1/fhe/health` that confirms the TFHE_Library server module is loaded, the parameter set is consistent with the current protocol version, and the Homomorphic_Evaluator is operational.

---

### Requirement 10: Performance Constraints and Graceful Degradation

**User Story:** As an institutional client, I want the FHE privacy mode to operate within acceptable latency bounds, and I want clear feedback when those bounds cannot be met, so that I can make informed decisions about which events to analyze under privacy mode.

#### Acceptance Criteria

1. THE FHE_Client SHALL display a latency warning in the dashboard UI when the measured round-trip time (encrypt → transmit → evaluate → return → decrypt) for a single event exceeds 15 seconds.
2. WHEN the server-side homomorphic evaluation for a batch of events exceeds 60 seconds total, THE Translation_Registry SHALL return partial results for the events completed within the timeout and SHALL include an `incomplete: true` flag in the FHE_Result for events that did not complete.
3. THE FHE_Client SHALL support batch submission of up to 50 RawEvents in a single FHE_Envelope to amortize the cost of FHE_Context transmission. WHEN a batch exceeds 50 events, THE FHE_Client SHALL split it into sequential batches of at most 50 events.
4. WHERE the institutional client's browser does not support WebAssembly SIMD instructions, THE FHE_Client SHALL detect this at initialization and present an informational message explaining that Privacy_Mode requires SIMD support, without activating Privacy_Mode.
5. THE Translation_Registry SHALL expose a `GET /api/v1/fhe/capabilities` endpoint that returns the supported TFHE_Library version, maximum supported ciphertext count per envelope, and the list of supported `fhePolicy` operations, allowing the FHE_Client to validate compatibility before activating Privacy_Mode.

---

### Requirement 11: Security Invariants

**User Story:** As an institutional client, I want formal guarantees that the Open-Audit server cannot infer plaintext values from the data it processes, so that I can deploy the system in compliance with institutional data-handling policies.

#### Acceptance Criteria

1. THE Homomorphic_Evaluator SHALL never call any TFHE_Library decryption function during request processing. THE codebase SHALL enforce this via a TypeScript module boundary: the decryption API SHALL be absent from the server-side TFHE_Library import.
2. THE Translation_Registry SHALL not persist any Ciphertext, FHE_Context, or FHE_Envelope data to the database, Redis cache, log files, or any durable store. All processing SHALL be in-memory and scoped to the HTTP request lifecycle.
3. THE FHE_Client SHALL generate a fresh Key_Pair for each Privacy_Mode session. Key reuse across sessions SHALL not be supported.
4. IF the FHE_Envelope's replay-detection nonce has been seen within the last 5 minutes (tracked in a server-side in-memory set scoped to the Node.js process lifetime), THEN THE Translation_Registry SHALL reject the request with HTTP 409 and error code `FHE_REPLAY_DETECTED`. The system SHALL set this error code only when a replay is actually detected; normal requests with unseen nonces SHALL not receive this error code.
5. THE FHE_Client SHALL validate that the server's FHE_Context echo in the FHE_Result matches the FHE_Context the client originally transmitted, to detect man-in-the-middle modification of the public key. IF there is a mismatch, THE FHE_Client SHALL abort decryption and surface a security alert in the dashboard.
